# v1 → main 升级差异总结

对比范围：`v1`  vs `main` 分支
本文档只聚焦升级时**用户感知的破坏面**：CRD schema、Chart ConfigMap/values、关键环境变量。
代码内部重构（包名 `internal/constants` → `pkg/constants`、deepcopy 等）不在此列。

---

## 一、CRD 变化

控制器代码生成版本 `controller-gen` 从 `v0.16.4` 升到 `v0.20.0`，所有 CRD 文件头部都会刷新 annotation。
按 schema 实质变化分类如下：

### 1.1 新增 CRD

| CRD | Scope | 说明 |
|---|---|---|
| **`providerconfigs.tensor-fusion.ai`** | Cluster | 全新资源，按硬件厂商（NVIDIA / AMD / Ascend …）管理镜像、硬件 metadata、虚拟化模板、`devicePluginDetection`、`inUseResourceNames`。`spec.vendor` 为主键字段。`charts/tensor-fusion/templates/rbac.yaml` 与 `config/crd/kustomization.yaml` 已加入。 |

### 1.2 `GPUPool` / `TensorFusionCluster.spec.gpuPools[].spec`

`componentConfig` 字段被整体精简，重点为：

| 变更类型 | 字段 |
|---|---|
| **移除** | `componentConfig.client.embeddedModeImage` |
| **移除** | `componentConfig.client.remoteModeImage` |
| **移除** | `componentConfig.client.patchEmbeddedWorkerToPod` |
| **移除** | `componentConfig.client.patchToEmbeddedWorkerContainer` |
| **移除** | `componentConfig.nodeDiscovery`（整段去掉，含 image / podTemplate） |
| **新增** | `componentConfig.client.image`（替代上面两个 image，未匹配到 ProviderConfig 时使用） |
| **新增** | `nodeManagerConfig.defaultVendor`（默认 `NVIDIA`，单厂商模式生效） |
| **新增** | `nodeManagerConfig.multiVendorNodeSelector`（多厂商模式，map\<vendor, NodeSelector\>） |
| **结构** | `nodeManagerConfig` 由可选变成 **required** |

升级前必须人工删除任意残留的 `embeddedModeImage` / `remoteModeImage` / `nodeDiscovery`，否则 apply 时被严格 schema 拒绝。

### 1.3 `GPU` (`gpus.tensor-fusion.ai`) status

| 变更 | 字段 | 说明 |
|---|---|---|
| **移除** | `status.model` | 顶层 model 字段被搬到 `runningApps` 之外的其他结构 |
| **移除** | `status.nvLink`（含 `peers[].peerUUID/linkVersion/linkCount/bandwidthMBps` 等） | NVLink 拓扑结构被 `topology` 替换 |
| **新增** | `status.topology.peers[]` | 字段：`peerGPUUUID`、`tier`、`linkType`、`bandwidth`；为 `GPUNetworkTopologyAware` 调度插件提供归一化拓扑 |
| **新增** | `status.isolationMode` | 枚举 `shared` / `soft` / `hard` / `partitioned` |
| **新增** | `status.allocatedPartitions`（map\<partitionUUID, AllocatedPartition\>） | 记录已分配 partition（slot、isolationGroup、podUid 等），支持 MIG / Ascend 虚拟化 |
| **放宽** | `status.usedBy` | 去掉 enum 校验（v1 限制 `tensor-fusion` / `nvidia-device-plugin`） |

### 1.4 `TensorFusionWorkload` / `WorkloadProfile`

| 变更 | 字段 | 说明 |
|---|---|---|
| **新增** | `spec.gangScheduling.minMembers` (int32) + `spec.gangScheduling.timeout` (duration string) | gang 调度，0 表示禁用 |
| **新增** | `spec.partitionTemplateId` | 与 `isolation: partitioned` 配合使用，从 Pod annotation `tensor-fusion.ai/partition` 读取 |
| **改动** | autoscaling 默认百分位描述 | `targetCompute/VRAMPercentile` 默认 `0.95 → 0.9`；`upperBoundCompute/VRAMPercentile` 默认 `0.99 → 0.95`（仅默认值文案变化，字段不变） |

### 1.5 `SchedulingConfigTemplate`

仅 autoscaling percentile 默认值文档刷新，**无 schema 字段增减**。

### 1.6 其他 CRD

`gpunodes` / `gpunodeclaims` / `gpunodeclasses` / `gpuresourcequotas` / `tensorfusionconnections` 仅 `controller-gen` 注解版本号变化，无 schema 实质变化。

### 1.7 Gang 调度（新增能力，跨 CRD）

main 新增了**gang scheduling**——同一组 worker 要么同时被调度，要么集体等待，避免"部分 worker 起来霸占 GPU + 另一部分起不来卡死"。不是一张独立 CRD，**字段铺在 `WorkloadProfile` 和 `TensorFusionWorkload` 的 `spec.gangScheduling` 子对象里**：

```yaml
apiVersion: tensor-fusion.ai/v1
kind: TensorFusionWorkload         # WorkloadProfile 也一样
spec:
  replicas: 8
  gangScheduling:
    minMembers: 8                  # 必须同时调度满 8 个才放行；< 8 全部 Pending 等资源
    timeout: "10m"                 # 等不到就全部退避；不填 / "0s" = 无限等
  resources:
    requests: { tflops: 10, vram: 16Gi }
  ...
```

**字段语义（`api/v1/workloadprofile_types.go`）**：

| 字段 | 类型 | 含义 |
|---|---|---|
| `minMembers` | int32, `>= 0` | 至少同时调度多少个成员才放行；`0` 表示**禁用**，行为退化为普通逐个调度 |
| `timeout` | duration string (`"5m"` / `"1h"`) | 凑不齐 `minMembers` 时最多等多久；不填 / `"0s"` 表示无限等待 |

**gang group 名字是自动从 workload 命名空间+名字派生的**——`<namespace>/<workload-name>`，不需要用户手动写 group id。同一个 workload 下所有 worker pod 自动归到一组。

**实现位置**：
- 调度器插件：`internal/scheduler/gpuresources/gpuresources.go` 加了 `gangManager *gang.Manager`，挂在 `PreFilter`（提前判断能否调度）+ `Permit`（凑齐 minMembers 之前所有 pod 卡在 Permit 阶段）
- 包：`internal/gang/`
- Pod annotation（operator 自动注入，不需要用户手动加）：
  - `tensor-fusion.ai/gang-min-members` = 数字
  - `tensor-fusion.ai/gang-timeout` = duration

**与 K8s 原生调度的关系**：用的是 kube-scheduler framework 的 `Permit` 扩展点（在 Reserve 和 Bind 之间挂起），所以不需要外挂 Coscheduling / Volcano，**直接生效**——只要 `Workload` 跑在 `tensor-fusion-scheduler` 上（v1/main 都是默认值）。

**升级注意**：
- v1 上**完全没有 gang 调度**，存量 `WorkloadProfile` / `TensorFusionWorkload` 不带 `gangScheduling` 字段是合法的（`omitempty`），升级后行为不变。
- 一旦启用 `minMembers > 0`，**未满足时 Pod 会停留在 PodScheduled = False 状态**（kube 事件里能看到 `gang scheduling pre-filter` / "waiting for gang"），运维监控要相应豁免——不要把这类 Pending 误判成调度失败告警。
- 与 preempt（`dynamicConfig.preemptClusterWideFromEnv`）联动：高优先级 gang pod 会按整组抢占——抢占 1 个低优先级 pod 不够，必须**释放够 minMembers 的资源量**，否则不抢。这点要测一下你的优先级配置是否如预期。

## 二、Chart ConfigMap / 模板变化

`charts/tensor-fusion/templates/` 与 v1 比较：

| 模板 | 变化 |
|---|---|
| `node-overlay.yaml` | **新增**。若集群存在 Karpenter `NodeOverlay` CRD，下发 `tensor-fusion-overlay`，为 `tensor-fusion.ai/index_0..f` 各设容量 `36`，配合 GPU index 调度。 |
| `controller-deployment.yaml` | `INITIAL_GPU_NODE_LABEL_SELECTOR` 不再硬编码默认值 `nvidia.com/gpu.present=true`，完全取自 `Values.initialGpuNodeLabelSelector`（**默认值已挪到代码常量 `InitialGPUNodeSelector`**，但 chart 默认不再注入空时的兜底，部署时需显式赋值或依赖代码默认）。 |
| `rbac.yaml` | ClusterRole 加入 `providerconfigs`；对 `*` resources 的动词从 `get/list/watch` 放宽为 `*`（请评估你的 RBAC 限制是否能接受）。 |
| 其它 ConfigMap (`config.yaml` / `vector-config.yaml` / `gpu-public-gpu-info.yaml` / `greptime-standalone.yaml` / `alert-manager.yaml`) | **无 diff**。 |

### 2.1 `values.yaml` 实际差异

```diff
schedulerConfig.profiles[].plugins.GPUResourcesFit.args:
- enableNvlinkAware: false
- resourceScoreWeight: 0.7
- topologyScoreWeight: 0.3
- singleGpuProtectWeight: 0.6

dynamicConfig:
+ preemptClusterWideFromEnv: true
```

- **移除** `GPUResourcesFit` 插件下 NVLink-aware 相关 4 个权重参数（`enableNvlinkAware` / `resourceScoreWeight` / `topologyScoreWeight` / `singleGpuProtectWeight`）。
- **新增** `dynamicConfig.preemptClusterWideFromEnv: true`，控制是否从环境变量读取跨租户抢占开关。

### 2.2 `GPUNetworkTopologyAware` 调度插件配置（重点：内部语义彻底重写）

v1 阶段 `GPUNetworkTopologyAware` 只是个空壳：仅注册 `Filter` 并永远返回 `Success`，唯一可配置参数是 `totalIntranetBandWidthGBps`（带宽兜底）。
main 上该插件被**完全重写**，现在同时实现 `PreFilter`、`Filter`、`Score`，基于新的 `GPU.status.topology.peers[].tier` 做归一化 tier 评估。

**旧 → 新 参数对照：**

| v1 args | main args | 说明 |
|---|---|---|
| `totalIntranetBandWidthGBps: 100` | **已废弃**（main 的 `GPUNetworkTopologyAwareConfig` 不再有此字段，老 chart 留下的会被静默忽略） | 升级时应删除 |
| — | `mode: soft \| hard` | `hard` 拒绝不满足拓扑约束的节点；`soft`（默认）只参与打分 |
| — | `topologySource: auto \| numa \| vendor` | `auto`（默认）优先用厂商互连拓扑，回退 NUMA；可强制其一 |
| — | `maxAllowedTier: 1` | 允许的最大 tier（数字越小越近），默认 `1`（同 NUMA / 同 NVLink 组）；`hard` 模式下超出会被拒 |
| — | `unknownTopologyPolicy: treat-as-worst \| reject` | GPU 未上报 topology 时的处理策略，默认 `treat-as-worst` |
| — | `preferLeastDamage: true` | 单 GPU 请求时优先选不破坏已存在高质量拓扑簇的 GPU，默认 `true` |

**main 的 chart 默认值（`values.yaml` 第 224-253 行）只在 `filter` 阶段启用了该插件，没有在 `score` 阶段启用。**
也就是说升级后默认行为是：
- 拓扑约束作硬过滤（`mode` 默认 `soft`，所以实际上不会拒掉节点）；
- 不参与打分，所有节点拓扑得分相同。

**要让 GPUNetworkTopologyAware 真正生效**，需要在 ConfigMap（`schedulerConfig.profiles[].plugins`）里同步两处：

```yaml
schedulerConfig:
  profiles:
    - schedulerName: tensor-fusion-scheduler
      plugins:
        filter:
          enabled:
            - name: GPUResourcesFit
            - name: GPUNetworkTopologyAware
        score:
          enabled:
            - name: GPUResourcesFit
              weight: 5
            - name: GPUNetworkTopologyAware    # ★ chart 默认没启用，需要手动加
              weight: 3
        reserve:
          enabled:
            - name: GPUResourcesFit
      pluginConfig:
        - name: GPUResourcesFit
          args:
            maxWorkerPerNode: 256
            vramWeight: 0.7
            tflopsWeight: 0.3
        - name: GPUNetworkTopologyAware
          args:
            # ★ 旧字段 totalIntranetBandWidthGBps 已废弃，请删除
            # 注意：这里的 mode 只能在 soft / hard 二选一（拓扑是否做硬过滤），
            # 与 Pod 的 IsolationMode (shared / soft / hard / partitioned) 是两个完全独立的概念，
            # 见 internal/scheduler/gputopo/types.go:28-29。
            mode: soft                       # soft（默认，仅打分）/ hard（不满足直接拒掉节点）
            topologySource: auto             # auto / numa / vendor
            maxAllowedTier: 1                # 0=同 NVLink 组 1=同 NUMA 2=跨 NUMA 3=未知
            unknownTopologyPolicy: treat-as-worst   # 或 reject
            preferLeastDamage: true
```

**单 Pod 级覆盖（无需改 ConfigMap）：** main 新增 3 个 Pod annotation，按工作负载粒度临时覆盖：

| Annotation | 对应配置项 |
|---|---|
| `tensor-fusion.ai/require-gpu-topology` | 覆盖 `mode`（`true` ⇒ hard） |
| `tensor-fusion.ai/gpu-topology-max-tier` | 覆盖 `maxAllowedTier` |
| `tensor-fusion.ai/gpu-topology-source` | 覆盖 `topologySource` |

**前置条件：**
- 节点上的 GPU 必须上报新的 `status.topology.peers[].tier`（hypervisor 需要为升级后的版本，并允许 NUMA/厂商拓扑探测）；
- 老 GPU 资源没有 `topology` 字段时，行为遵循 `unknownTopologyPolicy`。

### 2.3 `ProviderConfig` 模板字段约定（MIG / NPU 切分必读）

升级到 main 后，**MIG 分区元数据必须建在 `ProviderConfig.spec.virtualizationTemplates` 里**（v1 完全没有这套抽象）。schema 在 `1.2` 里已经讲了，这里讲**填写约定 + 关键字段语义**——直接决定 MIG 创建能不能跑通。

#### 2.3.1 三个 id 字段，谁是谁

| 字段 | 例子 | 谁看 | 不能填错 |
|---|---|---|---|
| `metadata.name` | `nvidia-provider` | k8s / operator 内部资源名 | 集群内唯一即可 |
| `virtualizationTemplates[].id` | `nv-80g-1g-10gb` | **operator 内部**：`partitionTemplateRefs` 引用它，resolver 按它建 map（必须在 `virtualizationTemplates` 内**唯一**） | 不能和同一 ProviderConfig 内其他 template 撞 |
| `virtualizationTemplates[].nvmlProfileId` | `19` | **真正传给 NVML** 的整数 GI Profile ID，operator 通过 resolver 翻译后传给 vgpu-provider | 必须是 NVIDIA 官方约定的整数；可重复（A100 的 19=1g.10gb 和 A100-40GB 的 19=1g.5gb 是同一个数字，对应不同物理 profile） |
| `virtualizationTemplates[].name` | `1g.10gb` | **用户面**：用户 Pod annotation `tensor-fusion.ai/partition` 里填的友好名 | 由 NVIDIA `nvidia-smi mig -lgip` 输出决定 |

**这是 main 新加的能力**（详见 `api/v1/providerconfig_types.go` 里 `NVMLProfileID` 字段）：把"chart-side 标识"（`id`）和"vgpu-provider/NVML 实际收的整数"（`nvmlProfileId`）解耦，可以让 A100 / A30 等不同架构（NVML id 在不同卡上语义不同）**共存在一份 `ProviderConfig` 里**。

#### 2.3.2 `id` 命名规约（chart 默认值用的）

`nv-<family>-<profile>[-me]`：

| 部分 | 含义 |
|---|---|
| `nv-` | 厂商前缀，预留 amd-/intel- |
| `<family>` | "整卡显存"标签：`80g` / `40g` / `a30`，未来加 `h100-94g` / `h200` / `b200` |
| `<profile>` | MIG profile 名，点→破折号保 DNS-1123 安全：`1g.10gb` → `1g-10gb` |
| `-me`（可选） | media engine 变体（singleton，单卡同时只能有 1 个） |

例子：`nv-80g-1g-10gb-me` = A100/H100 80GB 的 `1g.10gb+me` profile（NVML id 20）。

#### 2.3.3 hardwareMetadata.partitionTemplateRefs 必须填

每张 MIG-capable 卡的 hw entry 必须**显式列出**它支持的 template id：

```yaml
hardwareMetadata:
  - model: A100_SXM_80G
    partitionTemplateRefs:                       # ★ 缺这一行 → MIG 完全不工作
      - nv-80g-1g-10gb
      - nv-80g-1g-10gb-me
      - nv-80g-1g-20gb
      - nv-80g-2g-20gb
      - nv-80g-3g-40gb
      - nv-80g-4g-40gb
      - nv-80g-7g-80gb
```

resolver 走"模型先匹配 hw entry → 走该 hw 的 partitionTemplateRefs → 在全局 templateByID map 里查"，**不在 partitionTemplateRefs 里的 template 这张卡用不上**。

#### 2.3.4 同 vendor 单 ProviderConfig 的硬约束

`internal/provider/manager.go:115` 里 `m.providers[vendor]` 按 vendor 字符串去重——**集群内 `spec.vendor: NVIDIA` 的 ProviderConfig 只能存在一份**，多份后加载的会覆盖前一份。

跨 vendor（NVIDIA / Ascend / AMD）通过 `GPUPool.nodeManagerConfig.multiVendorNodeSelector` 按 k8s node label 自动选 vendor，可以同集群跑多种厂商；**但同 vendor 内多型号必须挤进一份 CR**——这就是为啥 chart 默认值用了"统一 vendor + 多 family template + nvmlProfileId 解耦"的设计。

#### 2.3.5 Hypervisor 重启风险

**任何一次 `ProviderConfig` CR 变更**（即使只改 description）→ `providerconfig_controller` 会列出全集群所有匹配 vendor 的 GPUNode，**逐个 `kubectl delete` 它的 hypervisor pod 让它重建**（`internal/controller/providerconfig_controller.go:83-128`）。运维流水线注意：
- Worker pod 不受直接影响（worker 在自己 pod 里跑），但**新 worker 创建会等 hypervisor 起回来**
- 现存 MIG GpuInstance **不会被销毁**（NVML 状态在硬件里，不在 hypervisor 内存里），但**新 partition 创建会停顿**
- 不要在生产高峰期频繁动 ProviderConfig；非紧急改动建议走变更窗口

#### 2.3.6 用户 Pod annotation 怎么写

`tensor-fusion.ai/partition` 接受**多种等价写法**（`internal/config/template_identifier.go` 做规范化）：

| Pod annotation | 命中机制 | 推荐度 |
|---|---|---|
| `"1g.10gb"` | 按 `name` 命中 | ★★★ 推荐 |
| `"mig-1g.10gb"` / `"mig-1g-10gb"` / `"MIG 1g.10gb"` | 规范化后命中 `name` | ★★ 兼容 |
| `"nv-80g-1g-10gb"`（chart-side id） | 按 `id` 直接命中 | ★ 不推荐（暴露 chart 实现细节） |
| `"19"`（NVML 数字） | **scheduler 侧不识别**（PartitionTemplateInfo 不携带 nvmlProfileId），调度会报 `partition template 19 not found` | ✗ 不可用 |

> **注意**：main 上 `tensor-fusion.ai/partition` 只接受 friendly name / chart-side id，**不要写 NVML 数字 id**。NVML 数字 id 仅在 `ProviderConfig` 的 `nvmlProfileId` 字段里使用（chart → vgpu-provider 内部映射），用户 Pod annotation 一律走 friendly name（`"1g.10gb"`）。

## 三、环境变量变化

### 3.1 新增的环境变量

| Env | 出现在 | 用途 |
|---|---|---|
| `TF_SCHEDULER_PREEMPT_CLUSTER_WIDE` | scheduler/operator | 配合 `dynamicConfig.preemptClusterWideFromEnv=true` 使用：`"true"` 集群级抢占，`"false"` 同命名空间抢占 |
| `TF_LOG_PATH` | worker | worker 日志路径 |
| `TF_LOG_LEVEL` | worker | worker 日志级别 |
| `KUBELET_CHECKPOINT_PATH` (`HypervisorKubeletCheckpointPathEnv`) | hypervisor | 读取 kubelet checkpoint，配合 device-plugin 检测 |
| `TF_HARDWARE_VENDOR` (`TFHardwareVendorEnv`) | hypervisor / worker | 多厂商模式（NVIDIA / AMD / Ascend …）选择驱动栈，例如 Ascend 时会自动注入 `LD_LIBRARY_PATH` |
| `TF_ACCELERATOR_LIB_PATH` (`TFAcceleratorLibPathEnv`) | hypervisor | 加速卡用户态库目录 |

### 3.2 重命名 / 改值的环境变量

| v1 | main | 说明 |
|---|---|---|
| `TF_CUDA_MEMORY_LIMIT` | **`TF_GPU_MEMORY_LIMIT`**（main 上同时**保留** `TF_CUDA_MEMORY_LIMIT` 作为 legacy 别名） | hard 限制器显存阈值，新代码用 `TF_GPU_MEMORY_LIMIT`（常量 `HardMemLimiterEnv`）；为兼容老 worker 镜像，operator 在 `SetWorkerContainerSpec` 中同时注入 `LegacyHardMemLimiterEnv = "TF_CUDA_MEMORY_LIMIT"`（同一 MB 值）。**升级时无需手动改 worker pod template**：新老镜像各自读自己的 env 即可，详见 PR #686 (`b93f1b5`)。仅当你自建的 pod template 强行覆盖了 worker 容器的 env 列表时，需要保留这两个变量同步。 |
| `HypervisorDefaultPortNumber = 8000` | `= 8001` | hypervisor 默认端口改到 8001，影响 Service / NetworkPolicy / 自建探针 |

### 3.3 移除的环境变量（含相关容器）

| Env | 说明 |
|---|---|
| `NODE_DISCOVERY_REPORT_GPU_NODE` | 整个 node-discovery 容器被砍掉，相关 env 不再注入 |
| `HOSTNAME`（在 node-discovery 容器上下文） | 同上 |
| `NVIDIA_VISIBLE_DEVICES` 注入逻辑（node-discovery）| node-discovery 不复存在；该 env 在其它容器仍按 `nvidia.com/gpu.present` 类逻辑使用 |
| `TFContainerNameNodeDiscovery = "tensorfusion-node-discovery"` | 容器名常量删除；该 sidecar 与其 image、podTemplate、CRD 字段一并下线 |

### 3.4 隔离模式驱动的 hypervisor 自动重启（main 新增机制）

> 你提到的"加了一个用于监控 node、根据池化模式 (soft/hard) 变化触发 hypervisor 重启"的功能 —— 它确实是 main 新增的，
> 但**触发媒介不是环境变量，而是 K8s Node 标签 + hypervisor 容器 CLI 参数**，所以放在这一节单独说明，避免误解。

**v1 完全没有此机制**：v1 的 `node_controller.go` / `gpunode_controller.go` / `pkg/constants` 里都没有 isolation-mode 相关 label 或重启逻辑。

**main 工作原理（`pkg/constants/constants.go:333` + `internal/controller/node_controller.go` + `internal/controller/gpunode_controller.go`）：**

1. **触发源**：在 K8s Node 上打标签 `tensor-fusion.ai/isolationMode = soft | hard | shared | partitioned`（常量 `HypervisorIsolationModeLabel`）。
2. **同步层**：`node_controller.syncGPUNodeLabelsFromNode` 把该标签从 K8s Node 同步到对应的 `GPUNode` CR；标签被删则同步删除。
3. **重启层**：`gpunode_controller` 每轮 reconcile：
   - `getHypervisorIsolationModeFromNodeLabel(gpuNode)` 读取期望值；
   - 用 `applyHypervisorIsolationModeArg` 把 `--isolation-mode=<mode>` 注入 hypervisor 容器 `Args[]`；
   - `isHypervisorIsolationModeConfigured(currentPod, desired)` 比对现有 pod 的实际 CLI 参数；不一致就 `Delete(pod)`，下一轮自动重建为新模式（日志会打印 `hypervisor pod deleted due to isolation mode label change`）。
4. **作用范围**：只重启该节点上的 hypervisor pod，worker pod 不会被这条逻辑触发重启（worker 走 isolation 注解 + recreate 流程，独立的路径）。

**与 `TF_ISOLATION_MODE` 这个新 env 的区别（容易混淆，请单独记忆）：**

| 项 | `tensor-fusion.ai/isolationMode`（Node 标签） | `TF_ISOLATION_MODE`（新 env） |
|---|---|---|
| 位置 | K8s Node label → GPUNode label | 业务容器（worker / inject-lib）env |
| 作用对象 | hypervisor pod | C cuda_hook（soft 模式下 `LD_PRELOAD` 进来的 libcuda_limiter） |
| 触发什么 | hypervisor pod 自动 delete 重建 | 库内代码分支选择 |
| 是否驱动重启 | **是** | 否 |

**升级注意：**

- 默认情况下没有这个 label，hypervisor `--isolation-mode` 参数也不会被注入，行为兼容 v1。
- 一旦在节点上打了 `tensor-fusion.ai/isolationMode` 并随后修改/删除，**operator 会主动删 hypervisor pod**；如果你的运维流水线对 hypervisor 重启敏感，请把节点 label 变更纳入变更窗口管控。
- 同节点上跑 worker 的隔离模式（Pod annotation `tensor-fusion.ai/isolation`）与节点 label 不要混设：节点 label 是 hypervisor 级别，Pod annotation 是工作负载级别，二者不强约束一致，但不一致时 limiter 行为可能不符合预期。

### 3.5 新增的 Pod Annotation（虽不是 env，升级注意）

- `tensor-fusion.ai/partition` (`PartitionNameAnnotation`)
- `tensor-fusion.ai/partition-id` (`PartitionTemplateIDAnnotation`)
- `tensor-fusion.ai/gang-min-members`
- `tensor-fusion.ai/gang-timeout`

## 四、升级动作清单

1. **CRD：** 用 main 的 `charts/tensor-fusion/crds/*.yaml` 全量 `kubectl apply`（kubectl 会自动处理 schema 升级）。`providerconfigs` 是新增 CRD，必须先于 operator 启动。
2. **清理过期字段：** 检查现网 `GPUPool` / `TensorFusionCluster.spec.gpuPools` 中是否使用了
   `componentConfig.client.embeddedModeImage|remoteModeImage|patchEmbeddedWorkerToPod|patchToEmbeddedWorkerContainer`，
   或 `componentConfig.nodeDiscovery.*`。这些字段升级后会被 schema 拒绝，需要替换为新的 `client.image` + `ProviderConfig`。
3. **node-discovery DaemonSet/容器：** v1 部署中的 `tensorfusion-node-discovery` 容器/DaemonSet 与其 RBAC、ConfigMap mount 一并下线。
4. **环境变量：** hypervisor `:8000 → :8001`，探活/Service/NetworkPolicy 同步改。**`TF_CUDA_MEMORY_LIMIT` 不用手动改**——operator (#686 起) 会同时注入新旧两个 env，老镜像继续读 `TF_CUDA_MEMORY_LIMIT` 即可；只有当你自建 pod template 整体覆盖 worker env 列表时，才需要确保 `TF_CUDA_MEMORY_LIMIT` 与 `TF_GPU_MEMORY_LIMIT` 都在。
5. **抢占开关：** 如果业务对跨命名空间抢占敏感，在 operator/scheduler 上显式设置 `TF_SCHEDULER_PREEMPT_CLUSTER_WIDE=false`，并保持 `dynamicConfig.preemptClusterWideFromEnv: true`。
6. **RBAC：** 评估 `rbac.yaml` 中对 `*` resources 放宽到 `*` 动词的影响；若有最小权限需求，需要在 overlay 中收紧。
7. **多厂商：** 单 NVIDIA 集群保持默认即可（`nodeManagerConfig.defaultVendor: NVIDIA`）；混合集群需配置 `multiVendorNodeSelector` 与各厂商 `ProviderConfig`。
8. **MIG / partition annotation**（见 § 2.3.6）：业务 yaml 里 `tensor-fusion.ai/partition` 只写 friendly name（`"1g.10gb"` / `"2g.20gb"` …），main 上 scheduler 不接受裸 NVML 数字 id。NVML 数字 id 只在 chart 侧 `ProviderConfig.nvmlProfileId` 里出现，不要泄漏到用户 Pod annotation。
9. **Gang 调度**（见 § 1.7）：如果之前没在用 gang，存量 workload 不带 `spec.gangScheduling` 字段，升级后行为完全不变。监控侧把"`gang scheduling pre-filter` 导致的 Pending"豁免出告警，避免误判。
10. **ProviderConfig 变更频率管控**（见 § 2.3.5）：编辑生效后 operator 会**自动重启全集群匹配 vendor 的 hypervisor pod**——不要在生产高峰期改 ProviderConfig；非紧急改动走变更窗口。
11. **回归验证：** 复用 `upgrade-validation/v1-pods/` 与 `upgrade-validation/main-pods/` 内 `soft-local-pytorch.yaml`、`hard-local-pytorch.yaml` 对照跑通。MIG 场景额外新增一个 `partitioned-mig-pytorch.yaml` 验证 `tensor-fusion.ai/isolation: partitioned` + `tensor-fusion.ai/partition: 1g.10gb` 端到端。

## 五、v1 → main 接管行为（in-place 升级是否能 zero-downtime）

> 关键问题：升级到 main 后，**v1 时期已经在跑的 worker pod 会不会被正确认领？**
> 结论：**会**，前提是按 § 四步骤先把 CR 残留字段清理掉。下面把每类资源接管路径拆给你看，并列出可能的 panic / 静默失败点。

### 5.1 Worker Pod 接管（核心问题）

**main operator 启动时的一次性 takeover 流程**（`internal/gpuallocator/gpuallocator.go:1772` → `ReconcileAllocationState`）：

1. 拿 leader lock 之后，调 `InitGPUAndQuotaStore` 把所有 GPU CR 读进内存。
2. 调 `ReconcileAllocationState`（`reconcileWorkerOnce` 保证只执行一次）：
   - `kubectl get pods --selector=tensor-fusion.ai/component=worker` 列出**全部存量 worker pod**
   - 过滤条件：`spec.nodeName != ""` 且 `(deletionTimestamp == nil OR finalizer 还在)`
   - 对每个合格 pod 调 `ComposeAllocationRequest`，**从 Pod annotation 反推 `AllocRequest`**
   - 注册到 `uniqueAllocation[podUID]` + `podNamespaceNsToPodUID` + `nodeWorkerStore`
3. 重新计算每张 GPU 的 `Available = Capacity − Σ(registered AllocRequest)`，与 GPU.status.runningApps 对齐
4. 启动 informer，转为 event-driven 模式

**接管的契约**（v1 / main 完全一致）：

| 资源 / 字段 | v1 = main 一致？ |
|---|---|
| Label `tensor-fusion.ai/component: worker` | ✓ 同 |
| Label `tensor-fusion.ai/workload: <name>` | ✓ 同 |
| Label `tensor-fusion.ai/managed-by: tensor-fusion` | ✓ 同 |
| Finalizer `tensor-fusion.ai/finalizer` | ✓ 同 |
| Annotation `tensor-fusion.ai/tflops-request` / `vram-request` | ✓ 同 |
| Annotation `tensor-fusion.ai/gpu-count` / `gpu-indices` / `gpu-ids` | ✓ 同 |
| Annotation `tensor-fusion.ai/qos` / `vendor` / `gpu-model` / `gpupool` | ✓ 同 |
| Annotation `tensor-fusion.ai/isolation` (`soft` / `hard`) | ✓ 同（main 新增的 `shared` / `partitioned` 值 v1 不会有，不影响） |

**结论：v1 worker pod 不需要重启 / 重建**，main operator 上来就能完整认领它们的资源占用。

### 5.2 GPU CR 接管

main 多读了几个 v1 GPU CR 没有的 `status` 字段，但**全部做了 nil-safe 处理**（`internal/gpuallocator/gpuallocator.go:1983` `syncGPUMetadataAndStatusFromCluster`）：

| 字段 | v1 上的状态 | main 接管行为 |
|---|---|---|
| `status.capacity` / `status.available` | 已有 | 直接采用 |
| `status.runningApps` | 已有 | 与 takeover 重算结果对齐（diff 部分按差值修正） |
| `status.topology` | **不存在** | `syncGPUMetadataAndStatusFromCluster:1999-2003` 显式处理 `nil` 分支，等 hypervisor 上报后回填；空时 `GPUNetworkTopologyAware` 走 `unknownTopologyPolicy: treat-as-worst` |
| `status.isolationMode` | **不存在** | CRD schema `default: soft` 自动填充，调度按 soft 处理 |
| `status.allocatedPartitions` | **不存在** | allocator 内部维护，v1 集群没有 partitioned 工作负载所以也用不到 |

→ GPU CR 不用手动迁移。hypervisor 重启后下一轮 reconcile 会自动把 `topology` / `isolationMode` 填回去。

### 5.3 Hypervisor Pod —— 一定会 rolling restart

main 的 hypervisor pod 模板差异巨大（新 env、新 args、新 volume），`HypervisorTemplateHash(pool)`（`internal/utils/compose.go`）算出来的 hash 一定和 v1 不同。`gpunode_controller` 看到 hash 不匹配会**主动删除老 hypervisor pod，按新模板重建**。

**影响**：
- **Worker pod 不掉**（worker 在自己 pod 里，跟 hypervisor 是独立进程，RPC 短暂断开后会自动重连）
- **新 worker 创建会暂停**几秒（要等新 hypervisor 起来 + 建立 RPC 通道）
- **现有 MIG GpuInstance 不会被销毁**（NVML 状态在硬件里，跟 hypervisor pod 生命周期无关）
- 节点多的集群建议**串行重启 hypervisor**，看 `componentConfig.hypervisor.podTemplate` 是否配了 `RollingUpdateStrategy`

### 5.4 GPUPool / TensorFusionCluster CR —— **可能踩坑**

⚠️ **风险点 1：残留字段被 schema 拒绝**

升级 CRD 后，老 CR 里的下列字段**新 schema 不认**，写入会被拒绝（读取不会，但下次 update 时 webhook / API server 会卡）：

- `componentConfig.client.embeddedModeImage`
- `componentConfig.client.remoteModeImage`
- `componentConfig.client.patchEmbeddedWorkerToPod`
- `componentConfig.client.patchToEmbeddedWorkerContainer`
- `componentConfig.nodeDiscovery.*`

**应对**：升级 operator **之前**，先把这些字段从所有现存 GPUPool / TensorFusionCluster 里删掉（`kubectl edit`），换成 `componentConfig.client.image`（指向 `tensor-fusion-client` 镜像）+ 配套 `ProviderConfig`。

⚠️ **风险点 2：`NodeManagerConfig` 在 main 上是 CRD-required 但 Go 端仍是 nil-able pointer**

main CRD schema 把 `nodeManagerConfig` 加进了 `required: [...]` 列表（`gpupools.yaml`），但 Go struct 仍然是 `*NodeManagerConfig` 带 `omitempty`。这导致：

- **老 v1 GPUPool 不带 `nodeManagerConfig` → Go 端 `pool.Spec.NodeManagerConfig == nil`**
- 多处代码**直接解引用**（`gpupool_node_provision.go:24` / `gpupool_controller.go:144` / `gpunodeclaim_controller.go:84` / `node_controller.go:153/154/156/164/369`）—— **nil pointer dereference panic**

**应对**：实操中绝大多数 v1 部署都已经设了 `nodeManagerConfig.provisioningMode`（默认 AutoSelect），所以不会真触发。但为了保险，**升级前 grep 全集群**：

```bash
kubectl get gpupool -o json | jq '.items[] | select(.spec.nodeManagerConfig == null) | .metadata.name'
kubectl get tensorfusioncluster -o json | jq '.items[] | .spec.gpuPools[]? | select(.spec.nodeManagerConfig == null)'
```

输出任何名字都得先补上 `nodeManagerConfig.provisioningMode: AutoSelect` 再升级。

### 5.5 Workload / WorkloadProfile 接管

- v1 上的 `TensorFusionWorkload` / `WorkloadProfile` CR **不带** `gangScheduling` / `partitionTemplateId` 字段，main schema 上这两个字段都是 `+optional`，**没问题**。
- v1 上的 autoscaling 百分位默认值（`targetCompute/VRAMPercentile: 0.95` 等）若有显式设置，升级后**取值保留**（schema default 只在字段缺失时生效）。

### 5.6 ProviderConfig —— **必须手动创建**

main 启动后会去 `mgr.GetProvider(vendor)` 查 ProviderConfig，v1 集群里**没有这个 CR**，结果：

- `gpunode_controller.go:527-529` 打 `"provider config not found; injecting empty hardware metadata"` warning
- hypervisor 仍然能起，soft / hard / shared 隔离模式照常工作
- **MIG（partitioned 隔离）完全失能**——没 templates 派发

**应对**：升级 operator **之前或之后立刻** `kubectl apply -f charts/tensor-fusion/templates/provider-config-nvidia.yaml`（chart 默认开启）或自己手写 ProviderConfig CR。具体看 § 2.3。

### 5.7 总体升级顺序建议

为了把"现存 workload 0 中断 + 新 workload 不被卡"作为目标，建议步骤：

| Step | 动作 | 风险 |
|---|---|---|
| 1 | `kubectl get gpupool,tensorfusioncluster -o yaml` 全量备份 | — |
| 2 | grep 残留字段（`embeddedModeImage` / `nodeDiscovery` / 空 `nodeManagerConfig`），先 `kubectl edit` 清理 | 拒绝升级前发现，安全 |
| 3 | `kubectl apply -f charts/tensor-fusion/crds/*.yaml`（main CRD）—— `providerconfigs` 必须先建 | CRD 升级对已有 CR 不做强校验，安全 |
| 4 | `kubectl apply -f <provider-config-nvidia.yaml>` 或确保 chart `providerConfigs.nvidia.enabled: true` | 不影响现存 worker |
| 5 | `helm upgrade tensor-fusion ./charts/tensor-fusion`（main chart） | operator 升级 + hypervisor 自动 rolling restart |
| 6 | 监控 `kubectl get pod -A -l tensor-fusion.ai/component=worker` —— worker 数量应保持不变；`tensor-fusion.ai/component=hypervisor` —— 应该全部按节点逐个重启 | hypervisor 重启窗口期间，**新 worker 调度会等几秒**；现存 worker 不掉 |
| 7 | 跑 § 四第 11 条的回归验证 yaml | — |

**回滚条件**：如果新 operator 报 `nil pointer dereference` panic 或大量 `partition template not found`（说明业务侧误写了 NVML 数字 id，见 § 2.3.6），把 operator/scheduler Deployment 镜像回滚到 v1 即可。

**CRD 要不要一起回滚？** 不强制，但 main 和 v1 schema 不是简单超集关系——既有新增也有删除字段，两个方向都会触发 k8s structural-schema pruning，需要明确取舍：

**v1 → main 的 schema 差异（影响回退判断）：**

| 类型 | 字段 | 影响方向 |
|---|---|---|
| main 新增 | `providerconfigs` CRD（整个资源）<br>`gpu.status.allocatedPartitions / topology / isolationMode`<br>`tensorfusionworkload.spec.gangScheduling / partitionTemplateId`<br>`tensorfusionworkload.status.gang` | 回 v1 后，这些字段会被 v1 schema pruning 剪掉（spec 字段一并消失）；`providerconfigs` 整个 CRD 必须手动删 |
| main 删除 | `gpu.status.model / nvLink`（含整套 nvLink peer 拓扑）<br>`tensorfusioncluster.spec.gpuPools[].specTemplate.componentConfig.client.{embeddedModeImage,remoteModeImage,nodeDiscovery,patchEmbeddedWorkerToPod,patchToEmbeddedWorkerContainer}` | 留 main CRD 时，v1 controller 想写这些字段会被 pruning 静默丢弃；不报错，只是数据看不到 |
| schema 收紧 | `GPUPool.spec.nodeManagerConfig`：`omitempty` → `+required` | 任何跑过 v1 hypervisor 的集群该字段已填（否则选不出 vendor），实际不会卡 |

**回退选项：**

| 选项 | 做法 | 后果 |
|---|---|---|
| **A：只回 image，保留 main CRD（最快止血）** | 仅 `kubectl set image deployment/...` 回 v1 镜像 | v1 controller 重新写 `gpu.status.nvLink / model` 会被 main schema **静默剪掉**——v1 控制台/dashboard 看不到 NvLink 拓扑信息，但 hypervisor / scheduler 运行不受影响；`providerconfigs` CR 留在 etcd 里休眠；main-only 字段（`allocatedPartitions / topology / isolationMode / gangScheduling` …）保留不动。**推荐用于紧急回退。** |
| **B：image + CRD 都回 v1（最干净）** | A + `kubectl apply -f <v1 charts>/crds/` + `kubectl delete crd providerconfigs.tensor-fusion.ai` | v1 schema 接管：v1 的 `nvLink / model / embeddedModeImage` 等字段恢复可写；**代价**是 main-only 字段在下一次 CR 写入时被 v1 schema 剪掉——`gangScheduling` / `partitionTemplateId` 这种 **spec 字段会被静默丢弃**（如果业务 yaml 没明确写就无所谓；写过就等同 gang 调度被关掉）。**注意** `kubectl apply` v1 CRD **不会**自动删 main 新增的 `providerconfigs` CRD，必须显式 `kubectl delete crd`。 |

### 5.8 不会被接管 / 需要人工迁移的场景（已知 limitation）

| 场景 | 行为 | 应对 |
|---|---|---|
| 节点上跑过 v1 的 `tensorfusion-node-discovery` DaemonSet | main 不再下发，但老 DS 资源不会自动删 | 手动 `kubectl delete daemonset tensor-fusion-node-discovery` |
| 用 Karpenter 给 GPU 节点做弹性扩容 | TF worker pod 经 webhook 注入 `tensor-fusion.ai/index_X: 1` 资源请求；现有 GPU 节点上由 hypervisor 内嵌 device plugin 注册 `tensor-fusion.ai/index_0..f: 36` 到 `node.status.allocatable`。但 Karpenter 在判断"要不要扩容一台新节点"时看的是 **NodePool/NodeClass + NodeOverlay 声明的未来节点容量**——没有 NodeOverlay 声明这 16 个伪资源，Karpenter 算不出"未来这台节点能放下这个 worker"，pod 触发 scale-up 时会卡 Pending（事件：`no instance type satisfies resources`） | 只要你的 GPU 节点是 Karpenter 管的，就必须确保集群里**有一份匹配 GPU NodePool 的 NodeOverlay**，`spec.capacity` 下包含 `tensor-fusion.ai/index_0..f: 36` 这 16 个键。具体 yaml 见下方两段示例。GPU 节点不是 Karpenter 管的（手动建 / ASG / cluster-autoscaler）完全跳过。 |
